# Acceptance State Machine Requirements

## Problem

The workflow already separates acceptance, execution, and audit effects, but it still starts delegation too early. A test designer can see product intent before the main agent has frozen files, symbols, interfaces, algorithms, isolation, and concurrency decisions; nobody independently reviews whether those tests have valid oracles; and one combined orchestration reference spends context explaining all roles, especially a late auditor. The delivery loop must make plan and framework design the upstream source of truth, review acceptance quality before code execution, and load only the contract for the role that is actually running.

## Requirements

- **REQ-001 — Design-bound test-first acceptance:** after the main agent freezes the implementation design and scaffold but before behavior implementation starts, a fresh acceptance designer must translate the requirements and design contract into `Validation.md` and executable focused tests. Declared acceptance paths are its planned focus; it may add necessary adjacent test helpers or fixtures and must report them to the main, but never edits production behavior, Node state, or verdicts.
- **REQ-002 — Explicit attempt phases:** every active implementation delivery must expose one machine-readable acceptance phase and a deterministic next action. The state/event pair must have at most one valid transition.
- **REQ-003 — Automatic post-executor regression:** when an assigned executor exits, the parent orchestrator must submit that exit event to the state tool, which runs the current Node's declared regression contract immediately. Neither the executor nor the main agent may claim the test outcome.
- **REQ-004 — Deterministic repair boundary:** when the reviewed design and acceptance inputs are unchanged, a failed or timed-out focused regression must persist only a redacted failure class and return the repair/defer decision to the native main. If the main chooses repair, a fresh executor for the same Node is the legal repair route. If design or acceptance inputs drift, the machine must cancel that route and select `dispatch_acceptance_designer`. A passing regression selects `dispatch_auditor`.
- **REQ-005 — Receipt-bound independent audit:** after regression passes, a fresh read-only auditor examines the exact tested fingerprint. An audit failure routes back to a fresh repair executor; an audit pass is valid only while that regression fingerprint remains current.
- **REQ-006 — Automatic acceptance and completion:** for automated delivery Nodes, the state tool alone checks regression-mapped criteria, records audit acceptance, and performs the completed transition. Manual criterion checking and manual early completion must be rejected while the automated cycle is active.
- **REQ-007 — Idempotent and private events:** duplicate dispatch or exit events must not repeat regression or completion. Persist only bounded phase, attempt count, opaque dispatch identity, safe outcome class, timestamps, and digests; never persist prompts, command output, absolute paths, agent transcripts, machine identity, credentials, or backend runtime data.
- **REQ-008 — Host-honest orchestration:** Hooks may explain the next action but must not intercept tools, spawn models, or pretend to observe a child exit. The native parent agent dispatches fresh leaf agents and immediately feeds their completion event into the state tool. Unsupported hosts remain skill-only.
- **REQ-009 — Final validation through the same mechanism:** final validation runs its declared full regression once, then requires a fingerprint-bound fresh audit before automatic completion. A failure produces a deterministic repair-plan action rather than reopening completed implementation history.
- **REQ-010 — Main-agent design ownership:** before any delegated acceptance or implementation role runs, the native main agent must write the plan, choose the module and file boundaries, list every symbol to add, modify, or remove, define interface signatures and error contracts, and precreate the agreed framework or scaffold. These decisions are Plan artifacts, not executor discretion.
- **REQ-011 — Role-specific context isolation:** main orchestration, acceptance design, acceptance review, execution, and final audit each have a separate reference contract. A Hook may inject only bounded main-agent location, phase, next action, and one repository-relative reference pointer; a delegated role loads only its own contract and declared Plan artifacts.
- **REQ-012 — Independent acceptance review:** a fresh read-only acceptance reviewer, different from the acceptance designer and executor, must approve requirement coverage, observable oracles, negative cases, false-positive resistance, and design/test fingerprint consistency before `dispatch_executor` becomes valid.
- **REQ-013 — Thin final audit:** the final read-only auditor receives the tested fingerprint, changed paths, acceptance target, and a minimal PASS-or-findings contract. It is not taught general code-review technique and does not compensate for missing design or weak tests.
- **REQ-014 — Preparation fingerprint invalidation:** acceptance approval binds separate digests for the current requirements and design contract, the pre-execution scaffold, and the validation mapping plus declared tests. Before executor dispatch, a change to any of those inputs invalidates approval. Once execution starts, planned implementation and scaffold edits do not invalidate the reviewed acceptance contract; requirements, design, validation, or test drift still cancels execution, repair, regression, or audit proof and returns the Node to acceptance design.
- **REQ-015 — Explicit engineering decisions:** every implementation Node must record owned paths, symbol operations, interfaces and dependency direction, composition or inheritance choice, algorithms and data structures with complexity, state and cache ownership, isolation boundaries, and concurrency or serialization rules. Independent Nodes may run concurrently only when their dependencies and owned paths are disjoint; mutation of one Plan state remains serialized.
- **REQ-016 — Framework-owned Agent lifetime:** Better Plan must not poll, time, interrupt, or replace a delegated Agent. Dispatch lifetime, cancellation, and host-level timeout behavior belong exclusively to the native agent framework. Better Plan may evaluate a role result only after the correlated Agent-completion event arrives.
- **REQ-017 — Current install and Hook removal surface:** installation and diagnosis must require every current state-machine, role-router, and separated role-contract file. Obsolete artifacts are eliminated with a one-time removal scan rather than retained compatibility code or permanent legacy-name tests. A public Hook-only uninstall operation must remove only Better Plan-managed Codex and Claude lifecycle handlers while preserving skills, unrelated settings, and unrelated Hooks. Cursor remains skill-only, no tool-use event is configured, and every managed lifecycle handler has a bounded timeout of at most 30 seconds.

## Non-goals

- Do not make tests generated after implementation the primary acceptance contract.
- Do not ask an acceptance designer or executor to invent architecture that the Plan has not fixed.
- Do not co-locate or co-inject multiple delegated-role prompt templates.
- Do not let an executor or auditor mutate its own verdict fields.
- Do not use `PreToolUse`, `PostToolUse`, permission denial, or arbitrary command interception.
- Do not replay or persist raw regression output.
- Do not reopen terminal Nodes when a later repair requires new work.
- Do not treat a child response, an unrelated diff, or a stale installed file as proof of delivery.

## Acceptance target

For each implementation Node, the main agent first freezes a machine-readable design contract and scaffold. A fresh acceptance designer writes executable proof from that contract, and a different acceptance reviewer approves its current fingerprint. Only then may the state machine follow `dispatch executor → executor exit → automatic regression → repair or minimal audit → completed`. Unchanged reviewed inputs make a failure an implementation repair; changed reviewed inputs make it an acceptance redesign. Every point has one deterministic next action, isolated role context, and no manual acceptance checkbox operation.
